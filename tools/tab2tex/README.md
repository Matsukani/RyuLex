# tab2tex.py

`tab2tex.py` converts RyuTab-formatted Excel data into TeX source for the RyuTeX template.

It can generate:

- dictionary entries
- Japanese → Ryukyuan reverse index

The script reads a RyuTab workbook (`.xlsx`), applies project-specific settings from an `.ini` file, and writes TeX output files for use in RyuTeX.

---

## What it does

The converter generates:

- `\entry{...}` blocks for the main Ryukyuan → Japanese dictionary
- `\rentry{...}` blocks for the Japanese → Ryukyuan reverse index

The script assumes that source data contain **no raw TeX**.  
All TeX markup is generated by the converter itself, and TeX special characters are escaped systematically.

---

## Input

The script expects:

- a RyuTab Excel workbook
- a settings file (`.ini`) specifying:
  - workbook path
  - sheet names
  - header mapping
  - orthography profile
  - sorting options
  - output options

The main lexical sheet is required.  
An example-sentence sheet is optional.

---

## Output

Depending on command-line options, the script writes:

- `*_entries_YYMMDD.tex`
- `*_reverse_YYMMDD.tex`

The output filename is based on the source workbook name and the current date.

---

## Processing model

The converter treats different field types differently:

- **Ryukyuan fields**  
  escaped + orthography profile applied

- **Japanese prose fields**  
  escaped + embedded Ryukyuan (`〈...〉`) processed + ruby applied

- **Ruby fields**  
  escaped + ruby processing applied

- **Plain Japanese fields**  
  escaped only

---

## Markup conventions expected in RyuTab

The script recognizes the following structured markup in source data:

- `〈...〉`  
  embedded Ryukyuan inside Japanese text

- `｜...〔...〕` / `...〔...〕`  
  ruby notation in Japanese text

These are treated as markup conventions, not as literal characters.

---

## Sorting and grouping

Entry sorting and headword grouping do **not** use raw Unicode order.

Instead, the script uses a hard-coded gojūon-based collation.  
Before sorting and grouping, kana are normalized in two stages:

1. **Core kana cleaning**
   - remove whitespace
   - remove user-configurable ignorable characters
   - normalize small kana to full-size kana
   - remove voicing distinctions

2. **Script unification**
   - convert katakana to hiragana for sorting/grouping only

As a result, sorting and grouping intentionally conflate:

- hiragana and katakana
- small and full-size kana
- voiced / semi-voiced and unvoiced kana

Displayed headletters are derived separately from grouping keys:

- `auto` mode: script is preserved
- `hiragana` mode: headletters are normalized to hiragana

---

## Reverse index

The reverse index is generated from the `意味` and `読み` fields.

- multiple Japanese items are split by `。`
- parenthesized leading disambiguators are trimmed for sorting
- reverse-index headwords are deduplicated
- sorting is based on normalized reading

The script raises an error if segmentation of `意味` and `読み` does not match.

---

## Validation

The script assumes that RyuTab data have already been validated.

---

## Command-line usage

```bash
python tab2tex.py --settings ryutex-settings.ini --o-entries
python tab2tex.py --settings ryutex-settings.ini --o-reverse
python tab2tex.py --settings ryutex-settings.ini --o-entries --o-reverse
```

## Command-line options

- `--settings`  
  Path to the settings file  
  Default: `ryutex-settings.ini`

- `--o-entries`  
  Output dictionary entries

- `--o-reverse`  
  Output Japanese → Ryukyuan reverse index

---

## Requirements

- Python 3
- pandas
- openpyxl

If `pandas` is missing, the script exits with an installation message referring to `tab2tex_requirements.txt`.

---

## Notes

- Key–value pairs in TeX output are always braced (`key={value}`) to ensure stable parsing.
- The current collation logic is intentionally hard-coded.
- More specialized section systems would require a separate configurable collation design.
